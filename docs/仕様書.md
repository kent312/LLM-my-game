## 1. コンセプト

**「ルールはコードが守り、物語はAIが紡ぐ」1人用TRPG。**

プレイヤーは自由なテキスト入力で行動を宣言し、AIのゲームマスター（GM）が物語を描写する。ただし判定・戦闘・リソース管理などのゲームルールはすべてコード側で決定論的に処理し、AIは「結果の描写」だけを担当する。

### 1.1 差別化（vs サーガ＆シーカー）

| サガシカの課題 | 本作の方針 |
|---|---|
| 出力ごとの「インク」課金 | ローカルLLM同梱により**買い切り・追加課金なし** |
| ルールの裏付けがなくAI任せの展開 | 判定・戦闘・成長はコードが管理。AIはズルも忖度もできない |
| オンライン必須 | **完全オフライン動作**（デフォルトの同梱モデル使用時。外部AI接続は明示的なオプトイン。4.5参照） |
| 展開が発散して物語が締まらない | シナリオ構造（起承転結・クエスト目標）をコード側で進行管理 |

### 1.2 コアバリュー

1. **遊びやすさ最優先** — TRPG未経験者でも遊べる。ルールの複雑さはコードが吸収する
2. **公平なゲーム体験** — ダイスと数値はコードが処理し、結果は覆らない。緊張感のあるゲームにする
3. **無限のロールプレイ** — 行動宣言は自由入力。AIが世界の反応を生成する

## 3. ゲームデザイン

### 3.1 コアループ

```
シーン提示（AI描写）
  → プレイヤーが行動をテキスト入力
  → コードが行動を解析（AIの意図分類 → コード検証）
  → 判定が必要なら実行し、結果（成功/失敗/クリティカル等）を確定
  → 状態更新（HP・所持品・クエスト進行）→ セーブ    ※必ず描写より前（INV-6）
  → AIが確定した結果を物語として描写
  → 次のシーンへ
```

この順序（**状態更新・保存が描写生成に先行する**）は判定を伴う行動・伴わない行動の両方に適用する。描写の生成失敗・中断が判定結果や状態更新を失わせる経路を作ってはならない（constitution INV-6、5.1参照）。

**重要な原則**: AIの出力はゲーム状態を変更しない。状態変更はすべてコードが行い、AIには「確定した結果」を渡して描写させる。

これは判定を伴わない行動（移動・会話・アイテム使用）にも適用する。ダイスを振らない行動もコードがシナリオデータと照合して効果を確定・保存してから描写する（「描写だけされて状態が変わらない」「できないことをできたことにする」経路を作らない。実装仕様書 §7.1）。

### 3.2 ルールシステム（コード側）

判定方式の詳細は「付録A: 判定ルール詳細」を参照。要点:

- **判定方式**（決定）: **2d6 ＋ 能力値修正 ＋ スキルボーナス**、固定しきい値の5段階判定（10以上=完全成功 / 7〜9=部分成功 / 6以下=失敗、出目12=クリティカル / 出目2=ファンブル）
- **ダイスを振るのはプレイヤーだけ**: 敵の攻撃も「PCの回避/防御判定」として処理する（PbtA方式）。コードの判定処理が一本化され、プレイヤーは常に自分のダイスで運命が決まる
- **能力値**（決定）: 6能力（筋力・敏捷・体力・知力・判断・魅力）。修正値を直接持つ（－1〜＋3）
- **スキル**（決定）: 固定スキルリスト＋自由記述の得意分野を併用
- **キャラクター作成**: 自由記述（名前・外見・背景）＋ 能力値の割り振り ＋ スキル選択。自由記述部分はAIがロールプレイに反映
- **戦闘**（決定: 二段構え）
  - MVP: 戦闘も物語の一部として「判定＋描写」で処理する軽量型。ダイスロール演出とHP増減はあり、専用戦闘画面はなし
  - Phase 2: 専用モードのターン制戦闘に拡張（行動選択・敵HP・ターン管理）。軽量型のデータ構造（敵ステータス、行動意図分類）を流用できる設計にしておく
- **成長システム**: セッション終了時に経験点 → 能力値/スキル成長
- **リソース**: HP、所持金、インベントリ（アイテムはマスターデータで管理）

### 3.3 シナリオ構造（決定: ワンショット集）

- **ワンショットシナリオ集**: 1本30分〜2時間の独立シナリオ × 複数本
- キャラクターはシナリオをまたいで持ち越し可能（経験点・所持品を引き継ぎ、緩やかなキャンペーン感を出す）
- サンドボックス/長編キャンペーンは初期スコープ外（発散リスクと制作コストのため）

シナリオはデータ駆動（YAML/JSON）で定義: シーン、目標、NPC、判定ポイント、分岐条件。AIには現在シーンのコンテキストだけを渡す。

### 3.4 AIの担当範囲

| 担当 | 内容 |
|---|---|
| **AIがやること** | シーン描写、NPCの台詞、プレイヤー行動への世界の反応、判定結果の演出 |
| **AIがやらないこと** | ダイス判定、ダメージ計算、アイテム増減、シナリオ分岐の決定、勝敗判定 |
| **コードがAIから受け取るもの** | プレイヤー入力の構造化解析（何をしようとしているか＝意図分類） |

## 4. AI仕様

### 4.1 推論エンジン・モデル

- **推論エンジン**: llama.cpp（ライブラリとして組み込み）
- **同梱モデル**: Qwen3 系（Apache 2.0）を第一候補
  - 品質枠: Qwen3-8B（Q4量子化、約5GB）
  - 軽量枠: Qwen3-4B（Q4量子化、約3GB）
- **ファインチューニング**: 【要検討】TRPGリプレイ風の日本語データで LoRA チューニング（語り口の品質向上）。MVP後の改善項目でも可
- **外部接続（オプション、決定）**: 設定画面から外部ローカルLLM（Ollama / LM Studio 等）や各種クラウドAPI（OpenAI互換エンドポイント）を指定可能にする。デフォルトは同梱モデルで、外部接続はこだわり派向けの上級設定として提供。**データ送信の境界と同意フローは 4.5 に定める（必須要件）**

### 4.2 構造化出力

- **原則: 構造化出力は文法制約デコードで保証する。プロンプトでの「JSONで答えて」という依頼だけに頼ることは禁止**。小型ローカルモデルは厳密JSONを容易に壊すため、トークンレベルで構造を強制する
  - 同梱モデル: llama.cpp の **GBNF文法**（サンプリング時に文法外トークンを排除）
  - 外部API: OpenAI互換の Structured Outputs（`response_format: json_schema`, strict）が使える場合はそれを必須とする。使えないエンドポイントでは検証＋限定リトライで代替（実装仕様書 §6.1）
  - いずれの場合も、コード側のスキーマ再検証（INV-3）は省略しない（多層防御）
- プレイヤー入力の意図分類（例: `{"action_type": "attack", "target": "goblin", "method": "剣で斬りかかる"}`）→ コードが判定処理
- 描写生成は自由テキストだが、長さ・フォーマットをプロンプトで制御

### 4.3 コンテキスト管理

- コンテキスト長: 8K〜16Kトークンを想定
- **常時注入**: システムプロンプト（GM人格・出力規則）、キャラシート要約、現在シーンのシナリオデータ
- **履歴管理**: 直近Nターンの生ログ ＋ それ以前は自動要約（ローリングサマリー）
- セーブデータに要約とゲーム状態を保存（会話ログ全文は任意保存）。保存形式・破損対策は 5.1 を参照

### 4.4 ガードレール（Steam規約対応・必須）

Steam の Live-Generated Content 規約に対応するため:

1. **プロンプトレベル**: GMシステムプロンプトに禁止事項（性的コンテンツ、実在人物、差別表現等）を明記
2. **出力フィルタ**: NGワード/パターンの後処理チェック。引っかかった場合は再生成 or 定型文フォールバック
3. **入力フィルタ**: 明らかな脱獄・不適切要求の検知と拒否応答
4. **ライブAI生成のアダルト性的コンテンツは実装しない**（Steam規約で例外なく禁止）
5. ストアページで Live-Generated Content の申告を行う

### 4.5 外部接続モードのデータ境界（プライバシー・必須要件）

自由入力TRPGでは個人情報やセンシティブなロールプレイ内容が入力に混入しうるため、外部接続時のデータ取り扱いを以下の通り定める。

**接続モードの定義と保証**

| モード | 通信 | 保証 |
|---|---|---|
| 同梱モデル（デフォルト） | 一切の外部通信なし | 完全オフライン。ストア表記の「オフライン動作」はこのモードを指す |
| 外部ローカルLLM（Ollama等） | ユーザー指定のローカル/LANエンドポイントのみ | ゲームはインターネットへ送信しないが、指定エンドポイント（LAN先を含む）へのデータ送出が発生するため、**下記の共通必須要件（同意・表示等）の対象** |
| クラウドAPI | ユーザー指定の外部サービス | 共通必須要件＋クラウド固有要件をすべて満たした場合のみ有効化可能 |

**全外部接続（外部ローカルLLM・クラウドAPI）共通の必須要件**

constitution INV-4 の通り、外部AI接続は種別を問わず明示的なオプトインなしに有効化できない。誤設定やLAN先サービス経由でセンシティブなロールプレイ内容が意図せず流出しうるため、外部ローカルLLMも例外にしない。

1. **明示同意**: 初回有効化時に「送信されるデータの種類」「送信先（エンドポイントのアドレス）」「送信先の挙動・保存ポリシーはゲームの管理外であること」を提示し、明示的な同意を得る。同意なしでは接続設定を保存できない
2. **送信データの範囲**: 送信するのは推論に必要なコンテキストのみ（システムプロンプト、キャラシート要約、現在シーン情報、会話履歴/要約、プレイヤー入力）。セーブデータ全体、Steamアカウント情報、ハードウェア情報は送信しない
3. **利用中の表示**: 外部接続の使用中は画面上に常時インジケータを表示（「外部AI接続中」＋接続先種別）。同梱モデル（オフライン）との区別を曖昧にしない
4. **ログ**: ゲームが外部に送信するのは推論リクエストのみ。プレイログ・テレメトリの外部送信は行わない（クラッシュレポート等を将来入れる場合は別途オプトイン）
5. **通信失敗時のフォールバック**: タイムアウト/エラー時は (a) リトライ → (b) 同梱モデルへの一時切替を提案 → (c) 拒否時はその場で入力を保持したまま待機。**生成失敗でゲーム状態やダイス結果が失われないこと**（判定確定と描写生成のトランザクション分離。5.1参照）
6. **ガードレール**: 4.4 の入出力フィルタは外部接続時も同一に適用する（INV-8）

**クラウドAPI固有の追加要件**

7. **APIキーの保存**: OSの資格情報ストア（Windows Credential Manager 等）に保存。不可の場合はローカル設定ファイルに保存し、平文保存である旨を設定画面に明示。セーブデータ・エクスポートデータには含めない
8. **未成年・規約への配慮**: 外部サービスの利用はそのサービスの利用規約・年齢制限に従う旨を同意画面に明記

## 5. 技術スタック

- **ゲームエンジン**: **Godot 4**（決定）
  - オープンソース・ロイヤリティなし、テキスト主体のUIに十分、Steam / Steam Deck 対応実績あり
  - llama.cpp 統合は GDExtension（既存のものを評価 → 不足なら自作ラッパー）。Phase 0 で疎通検証
- **言語**: GDScript を基本、llama.cpp バインディング部は C++（GDExtension）。必要に応じて C# も選択肢
- **データ**: シナリオ・アイテム等は YAML/JSON のマスターデータ。セーブは JSON（詳細は 5.1）
- **ライセンス注意**: 同梱モデル（Apache 2.0）とllama.cpp（MIT）のライセンス表記をクレジットに含める

### 5.1 セーブデータ仕様

キャラクター・経験点・所持品をワンショット間で持ち越す設計のため、セーブの信頼性はゲーム体験の生命線。以下を必須要件とする。

**構造とバージョン管理**

- セーブJSONの先頭に `schema_version`（整数）と `game_version` を必ず持つ
- スキーマ変更時は**マイグレーション関数を版ごとに実装**（v1→v2→v3 と順次適用）。旧版セーブは読み込み時に自動変換し、変換前のファイルはバックアップとして残す
- ロード時にスキーマ検証（必須フィールド、型、値域）を行い、検証失敗は「破損」として復旧フローへ

**原子的保存と破損対策**

- 書き込みは **tmpファイルに全書き込み → flush → 原子的な上書き置換** で行う（書き込み途中のクラッシュで既存セーブを壊さない）。バックアップのローテーションは現行セーブの**コピー**で行い、処理のどの時点でクラッシュしても「有効なセーブ本体 or 有効なtmp」が必ず存在する順序とする（詳細プロトコルは実装仕様書 §8）
- ロード時、有効なtmpファイルが残っていれば置換直前のクラッシュとみなして復元する（確定済みの判定結果を巻き戻さない。INV-6）
- 各スロットは直近 **2世代のバックアップ**を保持。ロード時に最新が破損していれば自動的に前世代へフォールバックし、その旨をプレイヤーに通知
- チェックサム（ファイル末尾 or 別ファイル）で破損検知

**トランザクション境界（生成中断への耐性）**

- ゲーム状態の更新順序を「**判定確定 → 状態更新 → 保存 → AI描写生成**」と定義する。ダイス結果と状態変更は描写生成の**前**に確定・保存されるため、生成中にクラッシュ/強制終了してもダイス結果は失われない（振り直しによるチート・巻き戻しも防ぐ）
- 再開時、描写が未生成の確定済み判定があれば「確定結果から描写を再生成」して続行する

**要約と構造化状態の整合**

- セーブには (a) 構造化ゲーム状態（HP・所持品・進行フラグ）と (b) AI用ローリングサマリーの両方を含めるが、**真実は常に (a)**。両者が矛盾した場合は (a) を正とし、(b) は (a) から再構築（再要約）する
- (b) の破損・欠落はゲーム進行を止めない（描写の文脈が一時的に薄くなるだけに留める）

**その他**

- APIキー等の資格情報はセーブに含めない（4.5参照）
- Steamクラウドセーブ対応を前提に、セーブは1スロット1ファイル・相対参照のみで自己完結させる

## 6. UI/UX

- **基本画面**: チャットログ風のメインテキストエリア ＋ 入力欄
- **常時表示**: キャラクターステータス（HP・所持品）、現在の目標（クエストログ）
- **判定演出**: ダイスロールを視覚的に見せる（コードの判定が公正だと伝わる演出。差別化ポイント）
- **入力補助**: 自由入力に加え、状況に応じたサジェストボタン（「調べる」「話しかける」等）— TRPG未経験者の間口
- **生成待ち時間対策**: ストリーミング表示（1文字ずつ流す）でローカルLLMの生成速度を演出に変える
- ビジュアル: 【要検討】立ち絵・背景の有無。テキスト主体でも雰囲気作りの静止画は欲しい（人力制作 or 素材。AI生成アセットは申告が増えるため方針を決める）

| リスク | 対策 |
|---|---|
| 小型モデルの日本語品質が不足 | Phase 0 で早期検証。不足なら LoRA / 8B以上 / プロンプト工夫。最悪は外部API前提に方針転換 |

## 付録A: 判定ルール詳細

### A.1 基本判定

```
2d6 ＋ 能力値修正 ＋ スキルボーナス ＋ 状況修正
```

| 合計値 | 結果 | 物語上の扱い |
|---|---|---|
| 出目12（修正前） | **クリティカル** | 望んだ以上の成果。追加の利益を得る |
| 10以上 | **完全成功** | 望んだ通りの結果 |
| 7〜9 | **部分成功** | 成功するが代償・複雑化を伴う |
| 6以下 | **失敗** | 失敗し、状況が動く（単なる「何も起きない」にはしない） |
| 出目2（修正前） | **ファンブル** | 最悪の展開。ただしゲームオーバー直行にはしない |

- クリティカル/ファンブルは**修正を加える前の出目**で判定する（能力が高くてもファンブルはあり得る）
- しきい値（10/7）は常に固定。難しさは状況修正で表現する

**確率の目安**（部分成功以上になる確率 / 完全成功以上になる確率）:

| 合計修正 | 何かは起きる（7+） | 完全成功（10+） |
|---|---|---|
| ±0 | 58% | 17% |
| ＋1 | 72% | 28% |
| ＋2 | 83% | 42% |
| ＋3 | 92% | 58% |
| ＋4 | 97% | 72% |

### A.2 能力値（6種）

| 能力値 | 領域 | 判定の例 |
|---|---|---|
| **筋力** | 腕力・破壊・威圧的な身体行動 | 扉をこじ開ける、掴みかかる、白兵武器で戦う |
| **敏捷** | 素早さ・器用さ・隠密 | 回避する、忍び寄る、鍵を開ける、射撃 |
| **体力** | 頑健さ・持久力 | 毒に耐える、長距離走破、飲み比べ |
| **知力** | 知識・推理・記憶 | 手がかりの分析、伝承の想起、罠の仕組みの理解 |
| **判断** | 観察・直感・意志 | 異変への気づき、嘘の見抜き、恐怖への抵抗 |
| **魅力** | 説得・交渉・魅了 | 交渉、はったり、演説、なだめる |

- 各能力値は修正値そのものを持つ（**－1〜＋3**、作成時上限＋2）
- **キャラ作成時**: 全能力±0 から開始し、＋5点を割り振る。最大2つまで－1に下げて、その分を追加で割り振ってよい
- **派生値**: HP＝8＋体力×2、所持重量やその他派生はMVPでは導入しない

### A.3 スキル（固定＋自由記述の併用）

- **固定スキル**（暫定10種）: 運動 / 隠密 / 観察 / 知識 / 医術 / 説得 / 威圧 / 細工 / 生存 / 戦術
  - キャラ作成時に **2つ** 選択
- **自由記述の得意分野**: 「鍵開け」「古代史」「料理」「昆虫の知識」など、プレイヤーが **2つ** 自由に記述
- **効果**: 判定に関連するスキル/得意分野が1つでもあれば **＋1**（複数該当しても＋1まで。インフレ防止）

**自由記述のタグ正規化（信頼境界の維持のため必須）**

自由記述の意味的な関連判定をプレイ中のAI裁量に任せると「AIはズルできない」原則が崩れるため、以下の通りキャラ作成時にタグへ正規化し、プレイ中はコードの完全一致照合のみで処理する。

1. **タグ・タクソノミー**: マスターデータとして固定のスキルタグ体系（20〜30個。例: `knowledge.history`、`craft.lockpicking`、`social.performance`、`nature.animals`）を定義する。固定スキル10種もこの体系上のタグである
2. **作成時マッピング**: プレイヤーが得意分野を自由記述 → AIがタクソノミー内のタグ1〜2個へのマッピングを提案 → **プレイヤーがUIで確認・修正して確定**。確定後はキャラシートに `{label: "古代史", tags: ["knowledge.history"]}` の形で保存。ラベルは描写用、タグが判定用
3. **プレイ中の照合**: 意図分類でAIが返せるのは**タクソノミー内のタグIDのみ**（GBNFの列挙制約で強制）。コードはタグIDとキャラシートの完全一致でボーナスを確定する。プレイ中にAIが意味的な近さを判断する余地をなくす
4. **判定ログの透明性**: 判定ログに「隠密＋1（適用）」だけでなく「`craft.lockpicking` は今回の行動に不適用」のように**採用/不採用の理由を明示**する。プレイヤーが「なぜボーナスが付かないのか」を検証できる
5. **紛争解決**: マッピングに不満がある場合、キャラシート画面からいつでも再マッピング可能（能力値と違い、タグの付け替えは成長リソースを消費しない）

### A.4 状況修正と有利/不利

- **状況修正**: －2〜＋2。シナリオデータで定義するか、AIの意図分類が難易度タグ（easy / normal / hard）を提案し、コードが「easy＝＋1 / normal＝±0 / hard＝－1」に変換。－2/＋2 はシナリオデータ指定のみ（AI提案では使わせない）
- **有利（アドバンテージ）**: 3d6 を振って**高い方の2個**を採用。準備・道具・仲間の助けがあるとき
- **不利**: 3d6 を振って**低い方の2個**を採用。暗闇・負傷・時間切迫など
- 有利と不利が両方あれば相殺して通常の2d6

### A.5 戦闘への適用（MVP軽量版）

- 敵はデータとして **脅威度（＋0〜＋3）/ HP / 攻撃力 / 特徴タグ** を持つ
- **攻撃**: PCが筋力（白兵）or 敏捷（射撃等）で判定
  - クリティカル: ダメージ2倍
  - 完全成功: 武器ダメージを与える
  - 部分成功: ダメージを与えるが代償（反撃被弾 / 武器落とし / 位置悪化 など。シナリオデータの候補から選択）
  - 失敗: 敵の攻撃が通る（敵攻撃力ぶんのダメージ）
- **敵からの攻撃**: PCの敏捷（回避）or 体力（受け）判定として処理。敵の脅威度を状況修正としてマイナス適用
- **HP0**: 即死ではなく「行動不能＋シナリオ上の危機」に移行（ワンショットでも救済ルートを用意）

### A.6 成長（ワンショット間の持ち越し）

- シナリオクリアで経験点を獲得
- 経験点の使途: 能力値＋1（上限＋3）/ 固定スキル追加 / 得意分野追加
- 成長ペース目安: 2〜3本クリアで1回成長

### A.7 判定処理とAIの役割分担（再掲・具体化）

```
プレイヤー入力「衛兵に気づかれないよう物陰を伝って進む」
  → AI意図分類: {ability: "敏捷", skill_tags: ["skill.stealth"], difficulty: "normal"}
     （skill_tags はタクソノミー内のタグIDのみ。GBNFで列挙制約）
  → コード: キャラシートとタグ完全一致照合 → 敏捷＋2、隠密（skill.stealth）所持＋1 → 2d6＋3
  → ダイスロール演出: 出目 4＋2 ＝ 6 → 合計9 → 部分成功
  → コード → AIへ: 「部分成功。進めたが代償: 候補[物音を立てる/持ち物を落とす]」
  → AI描写: 「あなたは影から影へ渡り歩く。だが最後の曲がり角で、
     腰の革袋が壁に擦れて硬い音を立てた——衛兵の足音が止まる。」
```

- ダイスの乱数、しきい値比較、ボーナス計算、HP増減はすべてコード
- AIが触れるのは「意図分類の提案」と「確定結果の描写」のみ。AIの提案（スキル候補・難易度タグ）は必ずコードが検証・クランプする
