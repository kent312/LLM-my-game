## 4. コアデータモデル

### 4.1 CharacterSheet（JSONスキーマ相当）

```jsonc
{
  "name": "string",
  "description": "string",          // 外見・背景の自由記述（AI描写用）
  "abilities": { "STR": 0, "DEX": 2, "CON": 1, "INT": 0, "WIS": 1, "CHA": 0 },
  "hp": { "current": 10, "max": 10 },   // max = 8 + CON*2
  "skills": ["skill.stealth", "skill.perception"],       // 固定スキル: タグIDで保持
  "specialties": [                                        // 自由記述の得意分野（付録A.3）
    { "label": "古代史", "tags": ["knowledge.history"] },
    { "label": "鍵開け", "tags": ["craft.lockpicking"] }
  ],
  "xp": 0,
  "inventory": [ { "item_id": "torch", "count": 2 } ],
  "money": 30
}
```

### 4.2 GameState（真実の置き場。INV-7）

```jsonc
{
  "character": { /* CharacterSheet */ },
  "scenario_id": "sc001_goblin_cave",
  "scene_id": "scene_03",
  "flags": { "found_secret_door": true },     // シナリオ進行フラグ（bool/int/string）
```

### 4.3 スキルタグ・タクソノミー（初期版。`game/data/skill_tags.json`）

固定スキル10種＋自由記述マッピング用14種＝**24タグ**。タグの追加はマスターデータ変更のみで可能（ARCH-3）。**AIが返してよい値はこのファイルに存在するIDのみ**（INV-3）。

```
固定スキル: skill.athletics(運動) skill.stealth(隠密) skill.perception(観察)
  skill.lore(知識) skill.medicine(医術) skill.persuasion(説得)
  skill.intimidation(威圧) skill.craft(細工) skill.survival(生存) skill.tactics(戦術)
自由記述用: knowledge.history knowledge.religion knowledge.nature knowledge.occult
  knowledge.streetwise craft.lockpicking craft.cooking craft.mechanics
  social.performance social.deception social.etiquette
  nature.animals nature.plants body.acrobatics
```

各タグは `{ "id": "...", "label_ja": "...", "hint_ja": "適用場面の説明" }` を持つ。`hint_ja` は意図分類プロンプトとキャラ作成UIの双方で使う。

## 5. 判定エンジン（`game/core/judgment.gd`）

### 5.1 インターフェース

```gdscript
class_name Judgment

# リクエスト（意図分類の検証済み出力から構築）
class Request:
    var ability: Types.Ability
    var skill_tags: Array[String]      # 意図分類が提案→検証済みのタグID
    var situation_mod: int             # -2..+2（クランプ済み）
    var roll_mode: Types.RollMode

# 結果（これがそのまま判定ログとセーブに載る）
class Result:
    var dice: Array[int]               # 振った全ダイス（有利/不利なら3個）
    var kept: Array[int]               # 採用した2個
    var natural: int                   # kept の合計（クリ/ファンブル判定用）
    var ability_mod: int
    var skill_bonus: int               # 0 or 1
    var applied_tag: String            # 採用したタグID（なければ ""）
    var rejected_tags: Array[String]   # 不採用タグ（ログ透明性。ARCH-6）
    var situation_mod: int
    var total: int
    var tier: Types.ResultTier

static func resolve(req: Request, sheet: CharacterSheet, rng: RandomNumberGenerator) -> Result
```

### 5.2 アルゴリズム（この順序で実装する）

1. ダイスを振る: NORMAL=2d6 / ADVANTAGE=3d6上位2 / DISADVANTAGE=3d6下位2
2. `natural` = 採用2個の合計。**natural==12 → CRITICAL、natural==2 → FUMBLE**（修正適用前に確定。以降の手順をスキップしてよいが total は計算して記録する）
3. スキルボーナス: `req.skill_tags` のうちキャラシート（skills ∪ specialties[].tags）に**完全一致**するものが1つでもあれば +1。採用タグは一致した最初の1つ、残りは rejected に記録
4. `total = natural + ability_mod + skill_bonus + situation_mod`
5. tier: total >= 10 → SUCCESS / >= 7 → PARTIAL / それ以外 → FAILURE

### 5.3 テストベクタ（`tests/test_judgment.gd` に必ず実装）

| ケース | 入力 | 期待値 |
|---|---|---|
| 部分成功 | dice=[4,3], 敏捷+2, タグ一致なし, 状況0 | natural=7, total=9, PARTIAL |
| 完全成功 | dice=[4,3], 敏捷+2, skill.stealth一致(+1), 状況0 | total=10, SUCCESS |
| クリ優先 | dice=[6,6], 能力-1, 状況-2 | natural=12, CRITICAL（totalに関わらず） |
| ファンブル優先 | dice=[1,1], 能力+3, タグ+1, 状況+2 | natural=2, FUMBLE |
| 有利 | dice=[5,2,6]（ADVANTAGE） | kept=[6,5], natural=11 |
| 不利ファンブル | dice=[1,5,1]（DISADVANTAGE） | kept=[1,1], natural=2, FUMBLE |
| ボーナス非重複 | skill.stealth と body.acrobatics 両方一致 | skill_bonus=1, rejected に片方 |
| クランプ | situation_mod=+5 を渡す | Request構築時に+2へクランプ（またはassert） |

乱数は `RandomNumberGenerator` を注入しテストではシード固定。**AIコードパスを一切通らずに全テストが走ること**（ARCH-1の受け入れ条件）。

## 6. AI連携層

### 6.1 バックエンド抽象（`game/ai/llm_backend.gd`）

```gdscript
class_name LLMBackend  # ARCH-2: ゲームロジックはこの型しか知らない

signal token_streamed(text: String)
signal generation_finished(full_text: String)
signal generation_failed(error: LLMError)

func generate(prompt: String, opts: GenOpts) -> void  # 非同期。UIをブロックしない（ARCH-7）
func cancel() -> void
func is_available() -> bool

class GenOpts:
    var max_tokens: int = 400
    var temperature: float = 0.8
    var grammar: String = ""            # GBNF。空なら自由生成
    var json_schema: Dictionary = {}    # grammarと等価のJSONスキーマ（外部API用）
    var stop: Array[String] = []

func supports_constrained_output() -> bool  # トークンレベルの構造強制が可能か
```

実装: `backend_local.gd`（llama.cpp / GDExtension呼び出し）、`backend_openai.gd`（HTTPRequest, OpenAI互換 `/v1/chat/completions`, stream=true）、`backend_mock.gd`（固定応答・遅延シミュレート。テストと開発の既定）。

**構造化出力の強制手段（バックエンド別・必須）**

小型ローカルモデルはプロンプトで依頼しただけの厳密JSONを平気で壊す。**構造が必要な生成（意図分類等）でプロンプト指示のみに頼る実装は禁止**し、トークンレベルの制約デコードを一次防衛とする。

| バックエンド | 一次防衛（トークンレベル強制） | 備考 |
|---|---|---|
| backend_local | llama.cpp **GBNF**（サンプリング時に文法外トークンを排除） | 構造上パース失敗は起こり得ない。起きたら文法生成器のバグ（P0-3） |
| backend_openai | `response_format: {type: "json_schema", strict: true}`（Structured Outputs） | 接続テスト時に対応可否を検出し `supports_constrained_output()` に反映 |
| 非対応の外部エンドポイント | なし（**検証＋限定リトライで代替**: スキーマ検証失敗→エラー内容を添えて最大2回再生成→失敗なら §6.2 のフォールバック） | 設定画面に「このエンドポイントは構造化出力非対応のため分類が不安定になる可能性」を表示 |

いずれの経路でもコード側のスキーマ再検証（INV-3）は**省略しない**。制約デコードは「壊れたJSONが来ない」ことしか保証せず、「値が文脈として妥当」かの検証はコードの責務。

**フォールバック連鎖（4.5の実装）**: 外部接続失敗 → 3回リトライ（指数バックオフ）→ 「同梱モデルに切り替えますか」ダイアログ → 拒否なら入力保持のまま待機。この間ゲーム状態は変化しない（判定済みなら保存済み。INV-6）。

### 6.2 意図分類（`game/ai/intent_classifier.gd`）

- 入力: プレイヤー入力テキスト＋現在シーンの要約＋キャラの特技一覧（タグID＋hint）
- 出力: 温度0.1、**GBNF強制**のJSON:

```jsonc
{
  "action_type": "check | talk | move | item | attack | other",
  "ability": "STR | DEX | CON | INT | WIS | CHA",
  "skill_tags": ["skill.stealth"],      // 0〜2個。タクソノミーのIDのみ
  "target": "exit:scene_02",             // 行動対象。閉じたIDのみ（下記）。対象がなければ null
  "difficulty": "easy | normal | hard",
  "needs_roll": true,                    // 判定不要な行動（単なる会話等）は false
  "summary_ja": "物陰を伝い衛兵から隠れて進む"   // 判定ログ表示用の行動要約
}
```

**`target` の閉じた語彙（ターンごとに動的生成）**: move/item/talk/attack はコード側の解決（§7.1）に、check はシナリオ効果の適用（§7.2）に対象IDが必要なため、`target` の許可値を**分類実行時に GameState から列挙して文法に埋め込む**。

- 許可値 = 現在シーンの `exits`（`exit:<scene_id>`）∪ 所持アイテム（`item:<item_id>`）∪ シーンのNPC（`npc:<npc_id>`）∪ 出現中の敵（`enemy:<enemy_id>`）∪ 現在シーンの `checks`（`check:<check_id>`。§9）∪ `null`
- `tools/gen_grammar.gd` はタクソノミー等の**静的部分**の文法/スキーマを生成し、`intent_classifier.gd` が実行時に `target` の enum 節を合成する（静的テンプレート＋動的enumの2段構成）
- **check の扱い**: action_type=check かつ needs_roll=true の入力が現在シーンの `checks` のいずれかに該当する場合は `check:<check_id>` を返す。該当しない場合は null（＝シナリオ定義外の**自由判定**。§7.2 の通り宣言的効果は適用されない）。checks の `trigger_hint` は許可値とともに分類プロンプトへ提示する — **どの check に該当するかの選択はAI、検証と効果適用はコード**。trigger_hint と入力の自由文照合をコードが行う実装は禁止（INV-3）
- 検証規則: action_type が move/item/talk/attack なのに `target` が null または許可値外 → 検証失敗として再生成1回 → それでも失敗なら**実行不可として確定**（「どこへ/何を、をもう少し具体的に」の確認応答。描写で進めない）。action_type=check で `target` が許可値外 → 再生成1回 → なお失敗なら **target=null（自由判定）として続行**（check では null を許容。安全側＝効果なしに倒す）。action_type=attack の `ability` は STR（白兵）/ DEX（射撃等）のみ許容（付録A.5。それ以外は検証失敗として通常の再生成→フォールバック経路）
- summary_ja 等の自由テキストをコードが再解釈して対象を推測する実装は禁止（INV-3）

- 構造強制: §6.1 の表に従い、backend_local では GBNF、外部APIでは json_schema を**必ず**指定する（プロンプト指示のみでのJSON生成は禁止）
- GBNF と JSONスキーマの enum 部分はビルド時に `skill_tags.json` から**生成**する（タグ追加時に両方が自動追随）。生成器は `tools/gen_grammar.gd` に置き、GBNF・json_schema を単一ソースから出力する
- **検証（INV-3）**: GBNFがあっても必ずコード側で再検証する — JSONパース → 各enum値の存在確認 → skill_tagsのタクソノミー照合 → difficultyのクランプ。検証失敗は1回だけ再生成、それでも失敗なら `action_type=other, needs_roll=false` として「もう少し具体的に何をするか教えてください」系の応答へフォールバック
- needs_roll=false の会話・移動・アイテム使用は判定（ダイス）を挟まないが、**必ず §7.1 の非判定アクション解決を通して状態を確定してから**描写生成へ進む（すべての入力にダイスを振らない。テンポ優先。ただし状態確定のスキップは不可）

### 6.3 描写生成（`game/ai/narrator.gd`）

- 入力（この順でプロンプト組み立て。トークン予算の目安）:
  1. システムプロンプト（GM人格・出力規則・禁止事項）〜600tok
  2. キャラシート要約 〜200tok
  3. 現在シーンのシナリオデータ（目標・NPC・雰囲気タグ）〜400tok
  4. ローリングサマリー 〜500tok
  5. 直近ログ（生テキスト、最大6ターン）〜1200tok
  6. 今回の確定判定結果（tier・行動要約・代償候補）〜150tok
- 出力: 自由テキスト、max 400tok、ストリーミング表示必須（ARCH-7）
- **描写はゲーム状態を変更しない（INV-1）**: 描写内でアイテム入手や大ダメージを「語って」も状態には何も起きない。状態に影響する展開はすべて判定結果と代償候補（シナリオデータ）から事前に決まっている
- 部分成功の代償: シーンデータの `complications` 配列からコードが1つ抽選し、確定情報としてプロンプトに渡す（AIに選ばせない）

### 6.4 ローリングサマリー（`game/ai/summarizer.gd`）

- 発火条件: 直近ログが6ターンを超えたとき、古い2ターンずつを要約へ畳み込む
- 要約は既存サマリー＋畳み込み対象を入力に再生成（max 300tok）
- 要約の破損・欠落はエラーにしない（INV-7: 構造化状態から最低限の文脈を再構築して続行）

### 6.5 プロンプトテンプレート管理

- `prompts/ja/*.txt` に配置（system_gm.txt / intent.txt / narrate.txt / summarize.txt / specialty_map.txt）。ハードコード禁止（ARCH-9）
- プレースホルダは `{{character_summary}}` 形式。テンプレートエンジンは String.format 相当の単純置換のみ（ロジックをテンプレートに入れない）

## 7. ターン状態機械（`game/flow/turn_machine.gd`）— INV-6 の実施点

```
IDLE
 → INPUT_RECEIVED（プレイヤー入力）
 → INPUT_FILTERING（ガードレール入力フィルタ。§10-1。ヒットしたら定型応答して IDLE へ。AIには一切渡さない）
 → CLASSIFYING（意図分類。AI呼び出し）
 → VALIDATING（コード検証。失敗→1回だけCLASSIFYINGへ戻る→それでも失敗はIDLEへフォールバック応答）
 → [needs_roll?]
    yes → ROLLING（判定エンジン実行＋ダイス演出）
        → COMMITTING（判定結果の効果適用（check=§7.2 / attack=§7.3）→ 状態更新 → セーブ書き込み完了まで待つ）
    no  → RESOLVING_ACTION（非判定アクション解決。§7.1）
        → COMMITTING（解決結果の状態更新 → セーブ完了まで待つ）
 → NARRATING（描写生成。生成テキストは必ずガードレールのセンテンスバッファ＋出力フィルタ（§10-2/3）を経由して表示）
 → FINALIZING（pending_narration をクリアして保存）
 → IDLE
```

**ガードレールはこの状態機械の必須経路である（INV-8）**: 入力フィルタは CLASSIFYING の前、出力フィルタ・センテンスバッファは NARRATING の表示前に、バックエンド種別（mock / local / openai）に関わらず常に適用される。turn_machine が guardrails を経由せずに分類・表示へ到達する経路を作ってはならない（PR-12 のテスト⑦⑧が規約）。

### 7.1 非判定アクションの解決（RESOLVING_ACTION）

移動・会話・アイテム使用は日常的に発生する行動であり、「描写だけされて状態が変わらない」経路を許すと、UI・セーブと物語が乖離する（ソフトロック・リロード時の矛盾の温床）。**needs_roll=false でも状態確定は必ずコードが行う**（INV-1/7 の適用点）。

`action_type` ごとの決定論的リゾルバを実装する:

入力は §6.2 の検証済み意図（`target` は許可値照合済みのID）。リゾルバは target を直接使い、自由テキストの再解釈は行わない（INV-3）。

| action_type | 検証（target と シナリオデータ照合） | 適用できる効果（ホワイトリスト） |
|---|---|---|
| move | `target`（`exit:<scene_id>`）が現在シーンの `exits` に存在し `condition` を満たすか | `scene_id` 更新、exit定義の `set_flags` |
| item | `target`（`item:<item_id>`）がインベントリに存在し、アイテム定義に使用効果があるか | アイテム定義の宣言的 `effect`（§9 の語彙のみ）＋消費 |
| talk | `target`（`npc:<npc_id>`）が現在シーンに存在するか | 会話済みフラグ、NPC `knows` の開示記録 |
| other / 雑談 | 下記の安全側検査を通過したもののみ | **状態変更なしを明示的に確定**（ActionResolution に `no_state_change: true` を記録） |

- 検証に失敗した行動（条件不成立の exit、持っていないアイテム等）は**実行不可として確定**し、AIには「できなかった」ことを確定情報として渡す。**描写で「できたことにする」経路を残さない**
- 解決結果は `ActionResolution`（適用した効果・失敗理由）として `pending_narration` に含め、判定結果（Judgment.Result）と同格に扱う — 描写生成・クラッシュ復元・判定ログの透明性（ARCH-6）すべて同じ扱い

**other 落ちの安全側処理（分類ミスで状態変更要求が素通りするのを防ぐ）**

意図分類の精度は実行時保証にならないため、`no_state_change: true` で進めてよい条件をコード側で限定する:

1. `other` に分類された入力に対し、コードが**状態変更語彙の軽量検出**（辞書＋正規表現。`game/data/state_verbs.json`: 拾う/取る/入手/飲む/食べる/使う/渡す/捨てる/移動/向かう/入る/出る 等）を行う
2. 検出ヒットなし → `no_state_change` として確定してよい（純粋な観察・雑談）
3. 検出ヒットあり → そのまま進めない。**再分類を1回**（ヒットした語彙をヒントとしてプロンプトに付与）→ move/item/talk に分類されれば通常処理 → なお `other` なら**実行不可として確定**し、「何を/どこへ、を具体的に」の確認応答を返す（曖昧なまま描写だけ進む経路を封鎖）
4. どの経路を通ったか（検出語・再分類の有無）を ActionResolution に記録する（ARCH-6）

### 7.2 判定結果のシナリオ効果適用（`game/flow/check_resolver.gd`。ROLLING → COMMITTING）

needs_roll=true の判定を「振って表示するだけ」で終わらせず、結果をシナリオ進行へ**決定論的に**接続する。接続キーは §6.2 の閉じた `check:<check_id>` のみで、trigger_hint 等の自由文照合は禁止（INV-3）:

- `target` が `check:<check_id>` の場合、該当 check の分岐を tier に応じてコードが適用する:

| tier | 適用する分岐 |
|---|---|
| CRITICAL / SUCCESS | `on_success` |
| PARTIAL | `on_partial`（`complication` 指定があればそれを適用。なければシーンの `complications` から抽選） |
| FAILURE / FUMBLE | `on_failure` |

- CRITICAL / FUMBLE の追加要素は**描写のみ**で表現し、適用効果は on_success / on_failure と同一（別効果が必要になったら `on_critical` / `on_fumble` をスキーマに追加する。現行スキーマでは定義しない）
- 分岐の効果は §9 の宣言的7語彙のみ（PR-05 の適用器を再利用）。未定義の分岐（例: `on_failure` なし）は「効果なし」として確定
- `target` が null の**自由判定**（シナリオ定義外）は宣言的効果を適用しない。判定結果の記録のみで進行フラグ等は変化しない — 「描写で進行したことにする」経路は §7.1 と同様に封鎖
- どの check のどの分岐を適用したか（自由判定なら「効果なし」）を ActionResolution と同格に `pending_narration` へ記録する（ARCH-6。判定ログから適用根拠が追える）

### 7.3 attack の解決（`game/core/combat.gd`。軽量戦闘＝仕様書 付録A.5）

- 入力: 検証済み意図（action_type=attack。`target`=`enemy:<enemy_id>` は `active_enemies` と照合済み、ability は STR / DEX のみ。§6.2）＋ Judgment.Result
- 武器ダメージ: インベントリ中 `damage` フィールドを持つアイテムの**最大値を自動採用**（なければ素手=1）。装備スロットは持たない（MVP）。どの武器を採用したかは記録に残す（ARCH-6）
- tier 適用表（効果適用は COMMITTING。§7.2 と同じ扱いで `pending_narration` に記録）:

| tier | 効果 |
|---|---|
| CRITICAL | 武器ダメージ×2 を敵HPへ |
| SUCCESS | 武器ダメージを敵HPへ |
| PARTIAL | 武器ダメージ＋代償（シーン `complications` から抽選。反撃被弾・武器落とし等はシナリオ作者が候補として定義する） |
| FAILURE / FUMBLE | 敵の `attack` ぶんのダメージをPCが受ける（FUMBLE の強調は描写のみ） |

- 敵HP0: `active_enemies` から除去し `flags["defeated_<enemy_id>"] = true` を設定（シナリオの `condition` から参照可能）
- PC HP0: 死亡ではなく**行動不能**として確定し、シナリオ直下の `on_defeat`（宣言的効果。§9。通常は救済シーンへの goto）をコードが適用する — 「HP0からの展開」をAIの描写に委ねない（INV-1）
- **敵からの攻撃に独立した機構は持たない（MVP）**: 敵の先制・奇襲・反撃はシナリオ `checks`（ability=DEX（回避）/ CON（受け）、脅威度は作者が `situation_mod` の負値として指定（既存クランプ -2..+2 に従う）、被弾は分岐の `damage`）で表現する。敵が自律的にターンを取る戦闘ループは Phase 2 の拡張（仕様書 3.3）であり、本節のデータ構造（enemies.json・active_enemies）を流用する

- **COMMITTING はセーブのフラッシュ完了をもって成功とする**。NARRATING 以降で何が起きてもダイス結果と状態は失われない（INV-6）
- `pending_narration` には確定判定結果一式（Judgment.Result＋行動要約＋代償）を格納。起動時にこれが非nullなら「前回の続きから描写を再生成しますか」フローに入る
- NARRATING 中のキャンセル（プレイヤーがスキップ）: 生成済みテキストを確定として FINALIZING へ。**再生成による内容の振り直しは提供しない**（INV-6の趣旨）

## 8. セーブ実装（`game/core/save_manager.gd`）

仕様書5.1をそのまま実装する。要点:

- パス: `user://saves/slot_{n}/save.json`、バックアップ `save.bak1.json` `save.bak2.json`
- 検証: 先頭フィールド `schema_version`（現在=1）、`checksum`（checksum以外のJSON正規化文字列のSHA-256）

**書き込みプロトコル（クラッシュセーフ。この順序を厳守）**

```
1. ローテーション: bak1 → bak2 に rename、save.json → bak1 に「コピー」
   （move ではなくコピー。save.json はこの時点で消えない）
2. save.tmp.json に全書き込み → flush（可能ならディレクトリの fsync も。
   Godot API で不足する場合は GDExtension 側で提供）
3. save.tmp.json → save.json へ原子的「上書き」置換
   （POSIX rename / Windows は ReplaceFile 相当。非原子的な delete→rename は禁止）
4. 置換成功後に save.tmp.json は存在しない（rename で消える）
```

**どの時点でクラッシュしても「有効な save.json」または「有効な save.tmp.json」が必ず存在する**ことがこのプロトコルの不変条件。手順1でセーブ本体を move する実装（save.json が存在しない瞬間を作る）は INV-6 違反として禁止する。

**ロードプロトコル**

```
1. save.tmp.json が存在し検証OK → 手順3の直前でクラッシュしたとみなし、
   save.json へ昇格（tmp の方が新しい確定状態）。検証NGなら tmp を削除
2. save.json を検証 → OKならロード
3. 破損なら bak1 → bak2 の順でフォールバック（その旨をプレイヤーに通知）
4. すべて破損なら通知してスロット選択へ
```
- マイグレーション: `migrations/v{n}_to_v{n+1}.gd` を順次適用。適用前ファイルは `save.pre_migration.json` として保持
- APIキー・外部接続設定はセーブに**含めない**（別ファイル `user://settings.json`、キーは可能ならOS資格情報ストア）

## 9. シナリオデータスキーマ（`game/data/scenarios/{id}/scenario.json`）

```jsonc
{
  "id": "sc001_goblin_cave",
  "title": "ゴブリン洞窟の遺失物",
  "estimated_minutes": 45,
  "intro_ja": "冒険の導入文…",
  "scenes": [
    {
      "id": "scene_01",
      "goal_ja": "洞窟の入口を見つける",           // クエストログ表示＆AIコンテキスト
      "mood_tags": ["forest", "dusk", "uneasy"],   // 描写の雰囲気指定
      "npcs": [ { "id": "guide", "name": "案内人ロズ", "persona_ja": "…", "knows": ["flag:map_hint"] } ],
      "checks": [                                   // このシーンで想定される判定の事前定義（任意）
        { "id": "check_find_entrance",              // 必須。シーン内で一意（§6.2 の check:<check_id> と §7.2 が参照）
          "trigger_hint": "入口を探す", "ability": "WIS", "situation_mod": 0,
          "on_success": { "set_flags": {"found_entrance": true}, "goto": "scene_02" },
          "on_partial":  { "set_flags": {"found_entrance": true}, "complication": "torch_lost", "goto": "scene_02" },
          "on_failure":  { "advance_clock": 1 } }
      ],
      "complications": [                            // 部分成功の代償候補（コードが抽選。§6.3）
        { "id": "torch_lost", "effect": { "remove_item": "torch" }, "hint_ja": "松明を落とす" },
        { "id": "noise", "effect": { "set_flags": {"alerted": true} }, "hint_ja": "物音で警戒される" }
      ],
      "exits": [ { "goto": "scene_02", "condition": "flag:found_entrance" } ],
      "enemies": ["goblin"]                         // 任意。シーン進入時に active_enemies へ展開（§7.3）
    }
  ],
  "enemies_ref": ["goblin", "goblin_boss"],
  "on_defeat": { "goto": "scene_rescue" },          // 必須。PC行動不能時の救済ルート（§7.3。宣言的効果のみ）
  "rewards": { "xp": 3, "money": 50 }
}
```

- `effect` はここに列挙した宣言的操作のみ（set_flags / remove_item / add_item / damage / heal / advance_clock / goto）。スクリプト実行は不可（ARCH-3）
- `condition` の構文は2種のみ: `flag:<name>`（フラグが truthy）/ `clock:<n>`（GameState.clock が n 以上）。式言語・任意評価は導入しない（ARCH-3）
- シナリオクリア時（最終シーンの目標達成）は `rewards` の xp / money をコードがキャラシートへ加算する。**経験点の使途（成長）は Phase 2**（仕様書ロードマップ）で、本キューでは付与・持ち越しまでを実装する
- `checks[].id` は必須かつシーン内で一意（ローダが検証）。意図分類の `check:<check_id>` 許可値と §7.2 の効果適用がこのIDを参照する
- シーン外の自由行動（定義externalな判定）は意図分類の結果から動的に判定を組み立てる。`checks` は「シナリオ作者が結果を制御したい要所」のみ定義すればよい。**自由判定は宣言的効果を持たない**（§7.2。進行に効かせたい判定は必ず checks に定義する）
- `on_defeat` はシナリオ直下に必須（ローダが検証。仕様書A.5「HP0は即死ではなく行動不能＋救済ルート」の実施点）。`scenes[].enemies` と `enemies_ref` の全IDは enemies.json に存在すること（ローダ検証）

### 9.1 敵データスキーマ（`game/data/enemies.json`）

```jsonc
{ "id": "goblin", "name_ja": "ゴブリン", "threat": 1, "hp": 4, "attack": 2, "traits": ["cowardly", "pack"] }
```

- `threat` は 0..3（仕様書A.5 の脅威度）。防御 checks の `situation_mod` へ反映する際は作者が負値で指定し、既存クランプ（-2..+2）に従う
- `traits` は描写用の特徴タグ。**ゲーム効果を持たない**（AIコンテキストにのみ渡す。効果が必要になったら宣言的語彙として別途定義する）

## 10. ガードレール（`game/ai/guardrails.gd`）— INV-8

1. **入力フィルタ**: 脱獄パターン（「システムプロンプトを無視」等の辞書＋正規表現）→ 検知したら意図分類に渡さず定型応答。辞書は `ng_words.json` の `jailbreak` セクション
2. **出力フィルタ**: 生成完了テキストに対し NGワード辞書（性的・差別・実在人物カテゴリ）を照合 → ヒットで**再生成1回**（temperatureを下げる）→ 再ヒットで定型文フォールバック（「GMは少し考え込んだ。…場面を仕切り直そう。」等）
3. ストリーミング中の暫定表示はセンテンス単位でバッファし、NGヒット時にそのセンテンスを表示前に破棄できる構造にする
4. システムプロンプトの禁止事項節は `prompts/ja/system_gm.txt` に含める（フィルタと二重化）
5. 全フィルタは同梱/外部接続/デバッグビルドの区別なく常時有効（INV-8）。設定でのOFFは提供しない

## 12. PR単位の実装計画

**この節が実装の作業キューである。** AIエージェント（Claude / Codex 等）は PR を1つ選び、記載の受け入れ条件をすべて満たして PR を作る。番号順が依存順（「依存」欄が空なら並行着手可）。

### 12.0 全PR共通ルール（Definition of Done）

1. ブランチ名は `pr/NN-短い名前`（下表の通り）。**1 PR = 1 作業単位**。スコープ外の変更を混ぜない
2. マージ前に **`bash tools/checks.sh` が exit 0** であること（テスト全件＋依存規則チェック。PR-00 で導入）
3. `game/core/` に触れる PR は対応する GUT テストを同一 PR に含める
4. 仕様と実装を乖離させない: 実装中に本書と矛盾が見つかったら、**コードを仕様に合わせる**か、合わせられない理由と本書の修正を PR に含める（constitution の INV には触れない）
5. PR マージ後、本書冒頭の「実装状況」を `PR-NN 完了（次: PR-MM）` に更新する（PR 内で更新してよい）
6. コミットメッセージは日本語

**マージ運用（決定）**: GitHub 上の Pull Request 作成は**必須としない**。`pr/NN` ブランチで作業し、checks.sh が exit 0 になったら main へマージして push してよい。その際、**マージコミット（`--no-ff` 推奨）のメッセージに、該当 PR の受け入れ条件を満たしたことのチェックリストを記載**する（GitHub PR を作る場合は PR 説明に同じものを貼る）。CI（PR-00 の ci.yml）は pull_request だけでなく **main への push でも実行**し、直接マージでも検証が走るようにする。

**実行コマンド（PR-00 完了後に有効）**

```bash
bash tools/setup.sh          # Godot本体とGUTを取得（冪等。GODOT_BIN があればスキップ）
bash tools/checks.sh         # マージゲート: テスト全件＋依存規則＋データ検証
# 個別実行:
"$GODOT_BIN" --headless --path . -s addons/gut/gut_cmdln.gd -gdir=res://tests -gexit          # 全テスト
"$GODOT_BIN" --headless --path . -s addons/gut/gut_cmdln.gd -gtest=res://tests/test_judgment.gd -gexit  # 単一

```

## 13. コーディング規約・運用

- GDScript: 公式スタイルガイド準拠。クラスは `class_name` 宣言、型注釈必須
- 命名: ファイル snake_case / クラス PascalCase / 定数 UPPER_SNAKE
- すべてのユーザー向け文字列は言語リソース経由（ARCH-9）。`tr()` を使う
- コミット: 日本語メッセージ、1PR以下の粒度。メッセージ末尾の Co-Authored-By 運用は継続
- テスト: `game/core/` への変更は対応テスト必須。AIプロンプト変更は P0 評価セットでの回帰確認を推奨
- 新規セッションでの作業開始手順: ① `docs/constitution.md` → ② 本書冒頭の「実装状況」で次の PR を確認 → ③ `git log --oneline -10` で直近の作業を確認 → ④ §12 の該当 PR の受け入れ条件に従って着手
